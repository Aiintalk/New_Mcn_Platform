# MCN 内容工作台 M1 · 部署与容量评估

> 面向运维/工程的部署与容量参考。结论先行:本系统是**内部工具**,瓶颈不在在线人数,而在**外部 AI/视频处理的并发与配额**及**视频存储增长**。服务器不是瓶颈,密钥池(配额轮换)+ 任务队列(削峰)+ OSS 直传与生命周期(控存储)才是关键。

---

## 一、容量结论(≤50 人起步)

- 日活约 20~30,峰值同时生成脚本 3~5 个;单次"人设仿写"≈ 视频解析(数秒~30s)+ AI 生成(20~60s),大部分时间在等外部 API,**不吃本机 CPU**。
- **能支撑多少人由外部服务配额决定**(AI 的 RPM/TPM、TikHub 的 QPS/积分、ASR 并发路数),不是服务器。
- 不需要 GPU(AI、ASR 均为外部 API)。

| 规模 | 注册 | 日活 | 峰值并发任务 | 服务器 |
|---|---|---|---|---|
| 起步 | ≤50 | 20~30 | 3~5 | 2核4G(最低)/ 4核8G(推荐) |
| 中等 | 50~200 | 50~100 | 8~15 | 4核8G + Redis 队列 |
| 较大 | 200~500 | 150~300 | 20~40 | 8核16G 或拆服务 |

---

## 二、两套真实环境

### 测试服(现有)
1 核 2 线程 / 3.4G 内存(可用 ~1.7G)/ 2G Swap / 79G 盘(可用 ~63G)/ Ubuntu 22.04 / 负载 0.07。

- 适用:开发联调、功能测试、≤10 人内测 —— **现在即可部署**。
- 不建议作 ≤50 人正式生产:单核是硬伤(后端与 PG 抢核)、内存偏小(依赖 Swap 兜底会变慢)。
- 别被"很闲(0.07)"误导:那是无人使用 + 未接外部 API 的状态。

### 正式服
约 **2 核 4 线程 / 6.8G 内存 / ~158G 盘**(测试服双倍)。

- ≤50 人生产 **达到推荐最低线,够用**。
- 两个必须守住的前提:**① PostgreSQL 尽量独立**(RDS 或单独进程限额),否则 2 核被应用+DB 争抢;**② 视频/文件一律走 OSS**,本地 158G 只放程序/日志/临时。
- 升级触发点:日活 >50 或峰值并发 >10 → 升 4核8G + 启用 Redis。

## 三、各能力承载与选型

| 能力 | 方案 | 本机压力 | 备注 |
|---|---|---|---|
| AI 脚本生成 | 第三方 API,后端转发 + 流式透传 | 极低 | 瓶颈是 TPM 与费用;队列 + 并发上限 + 密钥池轮换 |
| 文件上传(Word/附件) | OSS 前端签名直传 | 几乎为 0 | 不经过应用服务器 |
| 文件/图片存储 | OSS 对象存储,按量 | — | 单份 Word ~50KB、图片几百 KB,几万份仅数 GB |
| 视频上传 | OSS 前端签名直传 | 几乎为 0 | 单条 20~200MB,绝不落本地盘 |
| **视频存储(留档)** | OSS + **生命周期分层** | — | **最大成本变量**,见第四节 |
| 数据库 | PostgreSQL(正式建议独立/RDS) | 中 | 小内存需收敛参数 |

---

## 四、视频留档 —— 最大成本变量 ⚠️

已确认"视频需留档保存",该项**只增不减**,须提前规划。

- 估算(≤50 人):日活 25 × 每天 2 条 × ~100MB ≈ **5GB/天 → ~150GB/月 → ~1.8TB/年**,逐年累加。
- **必做:OSS 生命周期分层**——30 天转低频、90 天转归档。留档但分层,长期成本可压到约 1/3 甚至更低(归档取回有延迟,适合"很少回看但要留底")。
- 建议:仅长期保存"对标参考视频";运营过程素材设短保留期;`files` 表记录 OSS key + 大小 + 过期策略,便于算账与清理。
- 本地盘(测试 63G / 正式 158G)**绝不能存视频**,否则数天写满,且磁盘写满会导致 PG 直接挂。

## 五、月成本粗估(≤50 人,阿里云,含视频留档)

| 项 | 月成本(元,估) |
|---|---|
| ECS 4核8G(正式) | 200~350 |
| RDS PG 2核4G + 50G | 200~300 |
| OSS 存储(分层后,首年均摊) | 80~150,**逐月递增** |
| OSS 流量/请求 | 50~100 |
| 固定带宽 5M | 50~100 |
| **基础设施合计** | **约 600~1000 元/月起步**,随视频累积上涨 |
| **外部 API(AI+TikHub+ASR)** | **按量,可能 > 基础设施**,由调用量与单价决定,需单独测算 |

> 真正的大头通常是外部 API 调用费 + 视频存储/流量,而非服务器本身。

## 六、部署清单(测试机即可上手)

**1. 基础环境**
```bash
sudo apt update && sudo apt install -y nginx postgresql redis-server
sudo npm i -g pm2     # Node 用 LTS(20.x)
```

**2. PostgreSQL 内存收敛(小内存关键)** — `postgresql.conf`
```conf
shared_buffers = 256MB        # 小内存给 256~384MB,正式独立可 512MB~1G
effective_cache_size = 1GB
work_mem = 8MB
maintenance_work_mem = 64MB
max_connections = 40
```
`sudo systemctl restart postgresql`

**3. 后端并发限制(保护 CPU)**
- 任务队列最大并发:测试 3~5 / 正式 8~10,超出排队。
- 外部 API 超时:AI 60s、解析 30s,失败进重试,避免长请求堆住核。
- SSE 流式连接数随并发上限。

**4. PM2 守护**
```bash
pm2 start app.js --name mcn-api -i 1   # 单核 1 实例,勿开 cluster
pm2 save && pm2 startup
```

**5. Nginx 反代 + SSE** — 关键是关闭 buffering
```nginx
server {
  listen 80; server_name _;
  client_max_body_size 10m;          # 小文件;视频走 OSS 不经此
  location /api/ {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Connection '';
    proxy_buffering off;             # ★ SSE 流式必须关
    proxy_read_timeout 300s;
  }
  location / { root /var/www/mcn; try_files $uri /index.html; }
}
```

**6. 日志轮转(防写满磁盘)**
```bash
pm2 install pm2-logrotate
pm2 set pm2-logrotate:max_size 50M
pm2 set pm2-logrotate:retain 7
```
同时确认 PG / Nginx 日志的 logrotate 生效。

**7. OSS(测试也接,别落本地盘)**
- 前端签名直传;后端只发签名 URL;`files` 表存 key + 大小 + 过期。
- 测试桶生命周期"30 天自动删";生产桶按第四节分层留档。

**8. 安全基础**
- 防火墙仅开 80/443/22;PostgreSQL、Redis 仅绑 `127.0.0.1`,不对外。

## 七、测试 vs 正式 参数对照

| 项 | 测试(1核3.4G) | 正式(2核6.8G) |
|---|---|---|
| 应用实例 | PM2 单实例 | 单实例,DB 尽量独立 |
| 并发任务上限 | 3~5 | 8~10 |
| PG shared_buffers | 256MB | 512MB~1G(独立可更高) |
| Redis | 装了可不用 | 队列/多实例时启用 |
| 视频/文件 | OSS 测试桶(30天删) | OSS 生产桶(分层留档) |
| 定位 | 测试 / 预发 | ≤50 人生产 |

## 八、扩容路径(无需重构)

1. **50 → 200 人**:ECS 升 4核8G;PG 独立为 RDS;启用 Redis 做分布式任务队列;密钥池扩容(加 key)。
2. **200 → 500 人**:应用多实例 + 负载均衡(密钥池/队列移到 Redis);RDS 升配 + 读写分离评估;OSS 持续分层。
3. **若自建模型/自建 ASR**:需单独评估 GPU,属另一数量级,不在本评估范围。

---

## 九、关键提醒(一页纸)

- 不需要 GPU;服务器不是瓶颈,**配额 + 存储**才是。
- 视频**必须** OSS 直传 + 生命周期分层;本地盘永不存视频。
- 正式服把 **PG 独立**是 2 核机器最值得做的一件事。
- 留档是长期负债:立"保留策略 + 定期归档"的规矩。
- 测试机保留作预发环境,不浪费。
