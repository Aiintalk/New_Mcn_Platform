## 一句话定位

我们要给工具平台里的 AIGC 功能（先做千川仿写）建一套**「评价工具的工具」**：  
每次改完提示词，不再是靠少数几个人看几个案例主观判断好坏，而是用一批固定的测试样本、固定的评分标准，量化地告诉你**新版到底是变好了还是变差了，好在哪、差在哪**。

---

## 一、为什么要做（痛点）

现在迭代千川仿写这类 AIGC 工具时，普遍这么做：改一版提示词 → 拿几个经典案例跑一下 → 主观感觉「还不错」→ 上线。

三个问题：

1. **样本太少**：只看 3–5 个案例，不代表真实业务里五花八门的素材类型。改好了 A 场景，可能悄悄搞坏了 B 场景，没人发现。
2. **评分太主观**：每个人标准不一样，有人盯钩子，有人盯卖点，没有统一尺度。今天说好，明天换个人又说不行。
3. **迭代方向看不清**：改完一版，整体是变好还是变差？哪个维度进步了、哪个退步了？哪些类型的素材改善了、哪些恶化了？现在全靠猜。

一句话：**我们缺一把「能量化的尺子」，而且这把尺子要稳定、可复现、能长期追踪。**

---

## 二、核心设计（5 个概念，用人话讲）

整套体系围绕 5 个概念转。先记住这 5 个，后面的流程就通了。

| 概念 | 人话解释 | 例子 |
|------|---------|------|
| **测试集（test case）** | 一批固定不变的输入样本，覆盖各种场景 | 达人 A + 美妆产品 + 焦虑型卖点卡 |
| **版本快照（version）** | 某一版提示词+模型的完整「存档」，**改完就锁死**，不可再编辑 | v1.2-钩子强化（claude-sonnet + 某段 prompt） |
| **运行（run）** | 拿一个版本，跑一批测试样本，产出仿写结果 | 用 v1.2 跑「核心集」20 条样本 |
| **评分（score）** | 对每条产出，按几个维度打分（AI 先打，人再校准） | 文案质量 8 / 种草力 7 / 人设一致性 9 |
| **对比（compare）** | 把两次运行的分数摆一起，看差异 | v1.1 vs v1.2，整体 +0.5，但「焦虑型」样本降了 0.3 |

### 一期评分维度（先做这 3 个）

| 维度 | 看什么 | 默认权重 |
|------|--------|---------|
| **文案质量** | 钩子抓不抓人、顺不顺、信息够不够密 | 40% |
| **种草力** | 卖点讲清楚没、有没有行动引导 | 35% |
| **人设一致性** | 像「这个达人」会说的话吗，风格对不对 | 25% |

> 其余维度（卖点覆盖、合规安全、结构符合度等）记在 backlog，将来加。

### 评分怎么打：AI 初评 + 人工校准

AI 先按评分标准（rubric）给每条产出打分 + 写理由。然后人可以抽查、修正，**修正后的结果沉淀成基准**，越用越准。AI 评分不稳定，靠人工校准兜底。

## 三、怎么运转（一次完整流程）

```
开发者改了一版提示词
        ↓
在测试平台「创建版本快照」v1.2（锁死，不可改）
        ↓
触发一次运行：用 v1.2 跑「核心集」样本
        ↓
系统对每条产出：① 调 AI 仿写 ② 调 AI 按维度评分
        ↓
出报告：v1.2 的整体分、各维度分、每条样本明细
        ↓
和 v1.1 对比：整体涨/跌？哪个维度？哪类样本？
        ↓
涨了 → 这版提示词可以用；跌了 → 回滚或继续改
```

**触发方式**（一期）：
- 手动触发：人点一下，立刻跑。
- 版本创建后自动触发：新版本一建好，自动跑一次回归（可选开关）。
- 定时批量（夜间全量）：一期可后做，先保证前两个。

## 四、谁做什么（角色）

| | 运营 / 网红（最终用户） | 开发者/管理员（安雅、张翀、郜郜） |
|---|---|---|
| 用 AIGC 工具生成内容 | ✅ 照常用，不受影响 | — |
| 配置评分维度 / 标准 | — | ✅ |
| 创建版本快照 | — | ✅ |
| 录入 / 维护测试集 | （运营可协助） | ✅ |
| 触发运行、看报告、人工校准 | （运营可协助） | ✅ |
| 提交 case 反馈（"这条仿写不好"） | 二期才做 | 二期才做 |

**关键点**：一期**不动**现有千川仿写工具。运营该怎么用还怎么用，测试平台是独立的一块。最终用户的反馈入口（"一键提交到测试集"）是**二期**的事。

## 五、对现有系统的影响（重点说明）

这是大家最关心的：**会不会搞坏现在正在用的功能？**

**答案：一期不会。** 具体做法：

1. **代码独立**：全部新代码放 `backend/app/evaluation/`、`frontend/src/evaluation/`，不碰现有任何文件。
2. **数据库独立**：新增 `eval_` 开头的表，不改任何现有业务表。
3. **不动现有流程**：测试平台自己调 AI 生成，不调用现在的千川仿写接口。
4. **只读复用**：只「读」现有的达人数据、AI 适配器，不修改它们。

> 测试平台像在旁边搭了一个独立的「实验室」，不碰线上正在跑的「车间」。

**代价**：版本快照里的提示词，需要开发者手动从千川仿写配置复制过来（一期不做自动同步）。这是为了换「零影响」的有意取舍。

## 六、一期做什么 / 不做什么

### 做
- 被测对象：**千川仿写**（脚本文案）
- 评分维度：文案质量、种草力、人设一致性
- 测试集：人工录入（初期 10–30 条）
- 版本快照：手动创建，不可编辑
- 评分：AI 初评 + 人工校准
- 触发：手动 + 版本创建后自动
- 报告：单版本明细 + 两版本对比

### 不做（二期及以后）
- 最终用户 case 反馈入口
- 从生产数据自动回捞测试样本
- 与 CI/CD、部署流水线联动
- 视频成片、达人表现等其他对象的评测
- 评测平台独立成单独服务/仓库

## 七、未来怎么演进（路线图）

```
一期（现在）：千川仿写 + 3 维度 + 手动/自动触发，独立运行
   ↓
二期：用户反馈通路（"这条不好"→一键进测试集）；生产数据自动回捞
   ↓
三期：分层次回归（个例测试 → 批量回归 → 线上 case 回归 → 小流量观察）
   ↓
长期：复制到其他 AIGC 工具（千川复盘、直播脚本、视频成片…），可能独立成中台
```

**架构上提前留好了扩展口**（比如每个表都有 `tool_code` 字段、输入用灵活的 JSONB 存储），将来加新工具不用大改。

## 八、存储与成本（给个量级概念）

每次运行都把「输入 + 仿写结果 + 评分理由」完整存下来，方便长期追踪。

- 一次跑 100 条样本 ≈ 1–1.5 MB
- 一个场景一年 ≈ 2 GB；将来复制到 3–5 个场景，线性增长
- 应对：3 个月前的数据归档到冷存，热库只留近期

**结论**：量级不大，但会持续长，所以一期就把「输出归一化存储」「归档策略」设计好了，不堆到后面。

## 九、需要项目组确认的关键点（周会要拍板）

请大家在会上重点讨论这几个：

1. **3 个评分维度 + 权重**（40/35/25）认不认可？要不要调整或加减维度？
2. **AI 初评 + 人工校准**这个模式大家接不接受？人工校准谁来做、多久做一次？
3. **一期只做千川仿写**，范围 OK 吗？
4. **版本快照不可编辑**（改提示词要新建版本）这个约定大家能接受吗？
5. **测试集谁来维护**？初期那 10–30 条样本，谁来挑、从哪来？
6. **运营要不要参与**录入测试集、人工校准这些活？
