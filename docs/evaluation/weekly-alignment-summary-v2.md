## 一句话定位

我们要给工具平台里的 AIGC 功能（先做千川仿写）建一套**「评价工具的工具」**：  
每次改完提示词，不再是靠少数几个人看几个案例主观判断好坏，而是用一批固定的测试样本、固定的评分标准，量化地告诉你**新版到底是变好了还是变差了，好在哪、差在哪**。

---

## 一、为什么要做（痛点）

现在迭代千川仿写这类 AIGC 工具时，普遍这么做：改一版提示词 → 拿几个经典案例跑一下 → 主观感觉「还不错」→ 上线。

三个问题：

1. **样本太少**：只看 3–5 个案例，不代表真实业务里五花八门的素材类型。改好了 A 场景，可能悄悄搞坏了 B 场景，没人发现。
2. **评分太主观**：每个人标准不一样，有人盯钩子，有人盯卖点，没有统一尺度。
3. **迭代方向看不清**：改完一版，整体是变好还是变差？哪个维度进步了、哪个退步了？现在全靠猜。

一句话：**我们缺一把「能量化的尺子」，而且这把尺子要稳定、可复现、能长期追踪。**

## 二、核心概念（用人话讲）

| 概念 | 人话解释 | 例子 |
|------|---------|------|
| **测试集** | 一批固定不变的输入样本，覆盖各种场景 | 达人 A + 美妆产品 + 焦虑型卖点卡 |
| **版本快照** | 某一版提示词+模型的完整「存档」，**改完就锁死**，不可再编辑 | v1.2-钩子强化（某模型 + 某段 prompt） |
| **运行** | 拿一个版本，跑一批测试样本，产出仿写结果 | 用 v1.2 跑「核心集」20 条样本 |
| **评分** | 对每条产出，按几个维度打分（AI 先打，人再校准） | 文案质量 8 / 种草力 7 / 人设一致性 9 |
| **对比** | 把两次运行的分数摆一起，看差异 | v1.1 vs v1.2，整体 +0.5，但「焦虑型」样本降了 0.3 |
| **评测策略**（v2 新增） | 按业务/红人，把「用哪些测试样本 + 用哪套评分标准」**组合成一套方案** | 护肤业务一套、减肥业务一套 |

> 「评测策略」一期先用一个**默认方案**（全部样本 + 默认标准），结构留好，将来按业务/红人切换方案时不返工。

### 一期评分维度（先做这 3 个）

| 维度 | 看什么 | 默认权重 |
|------|--------|---------|
| **文案质量** | 钩子抓不抓人、顺不顺、信息够不够密 | 40% |
| **种草力** | 卖点讲清楚没、有没有行动引导 | 35% |
| **人设一致性** | 像「这个达人」会说的话吗，风格对不对 | 25% |

> 维度和权重**都由安雅定**（见第六节），上面是建议起步值。其余维度（卖点覆盖、合规安全等）记在 backlog，将来加。

### 评分怎么打

1. **AI 初评 + 人工校准**：AI 先按评分标准给每条产出打分 + 写理由；人抽查、修正，**修正结果沉淀成基准**，越用越准。
2. **分场景评分细则**（v2 新增）：同一个维度，不同品类可以有不同的打分标准。比如「文案质量」——美妆场景看「皮肤刺激/吸收」，减肥场景看「是否反弹/饮食注意」。一期先用一套通用标准，结构支持按品类细化。
3. **评委模型**（v2 新增）：给作品打分的那个 AI，**换一个评委，分就不一样**。所以每次评测要记录用了哪个评委（可复现），将来还能对比不同评委（文案类用 glm/kimi，视频类用 gemini/qwen 等）。一期固定一个评委。

## 三、怎么运转（一次完整流程）

```
开发者改了一版提示词
        ↓
在测试平台「创建版本快照」v1.2（锁死，不可改；配置自动从现有工具抠过来关联）
        ↓
触发一次运行：用 v1.2 + 默认评测策略，跑「核心集」样本
        ↓
系统对每条产出：① 调 AI 仿写 ② 调评委 AI 按维度评分
        ↓
出报告：v1.2 的整体分、各维度分、每条样本明细
        ↓
和 v1.1 对比：整体涨/跌？哪个维度？哪类样本？
        ↓
涨了 → 这版提示词可以用；跌了 → 回滚或继续改
```

**触发方式**（一期）：
- 手动触发：人点一下，立刻跑。
- 版本创建后自动触发：新版本一建好，自动跑一次回归（可选开关）。
- 定时批量（夜间全量）：一期可后做，先保证前两个。

## 五、对现有系统的影响（重点说明）

大家最关心的：**会不会搞坏现在正在用的功能？**

**答案：一期不会。** 具体做法：

1. **代码独立**：全部新代码放 `backend/app/evaluation/`、`frontend/src/evaluation/`，不碰现有任何文件。
2. **数据库独立**：新增 `eval_` 开头的表（v2 共 11 张），不改任何现有业务表。
3. **不动现有流程**：测试平台自己调 AI 生成，不调用现在的千川仿写接口。
4. **只读复用**：只「读」现有的达人数据、AI 适配器，不修改它们。
5. **不碰 ai_models / 凭证表**（v2 明确）：模型身份评测模块自己管，**完全不读不写**现有的模型表/凭证表，对现有业务模块零影响。

> 测试平台像在旁边搭了一个独立的「实验室」，不碰线上正在跑的「车间」。

**版本快照怎么来**（v2 改进）：不再靠开发者手动复制提示词——系统**自动从现有千川仿写配置抠出来关联**到版本快照（只读，不改现有配置），省事且可追溯。

## 六、安雅需要提供什么（一期千川仿写）

> 这节是请安雅定的事。完整填写模板见 **`docs/evaluation/scoring-alignment-anya.md`**（含每个维度的评分量化表）。

安雅只需定**三件事**（策略层、评委模型、适配器这些一期默认，都不用管）：

| 要定的 | 说明 | 建议起步值（安雅改） |
|--------|------|--------------------|
| **① 评分维度** | 评哪几个方向 | 文案质量 / 种草力 / 人设一致性（确认/增减） |
| **② 各维度权重** | 各占多少比例（加起来=100%） | 40% / 35% / 25% |
| **③ 评分量化表（rubric）** | 每个维度、每个分数档的具体打分标准 | 见下方样例 |

**评分量化表样例**（文案质量·通用场景，1–10 分、5 档；安雅细化/纠正）：

| 分数档 | 打分标准（安雅改） |
|--------|------------------|
| **10** | 开头 3 秒钩子极强，叙事流畅，卖点密度高不生硬，强代入感 |
| **8** | 钩子较抓人，叙事较流畅，卖点清楚，代入感尚可 |
| **6** | 有钩子但力度一般，叙事基本通顺，卖点提到但不突出 |
| **4** | 开头平淡，叙事生硬/跳跃，卖点堆砌 |
| **2** | 无明确钩子，叙事混乱，信息散 |

> 想做**分场景**（美妆/减肥各有不同标准）也行——同一个维度写多套量化表，系统按业务挑哪套。完整模板（含美妆/减肥变体、另两个维度）在 `scoring-alignment-anya.md`。

## 七、一期做什么 / 不做什么

### 做
- 被测对象：**千川仿写**（脚本文案）
- 评分维度：文案质量、种草力、人设一致性（安雅定权重/标准）
- 测试集：人工录入（初期 10–12 条，张翀提供、志伟录入）
- 版本快照：手动创建、不可编辑，配置自动关联
- 评分：AI 初评 + 人工校准；固定一个评委模型
- 评测策略：一条默认策略（结构预留，将来按业务/红人切换）
- 触发：手动 + 版本创建后自动
- 报告：单版本明细 + 两版本对比

### 不做（二期及以后）
- 最终用户 case 反馈入口
- 从生产数据自动回捞测试样本
- 分场景评分细则的切换 UI（结构预留，一期只用通用标准）
- 多评委对比、按内容类型选评委（评委候选池二期填）
- 与 CI/CD、部署流水线联动
- 视频成片、达人表现等其他对象的评测
- 评测平台独立成单独服务/仓库

## 八、未来怎么演进（路线图）

```
一期（现在）：千川仿写 + 3 维度 + 默认策略 + 手动/自动触发，独立运行
   ↓
二期：用户反馈通路；生产数据自动回捞；分场景评分切换；多评委对比
   ↓
三期：分层次回归（个例 → 批量 → 线上 case → 小流量观察）
   ↓
长期：复制到其他 AIGC 工具（直播间脚本仿写、价值观仿写、千川复盘、视频成片…），可能独立成中台
```

**架构上提前留好了扩展口**：按业务组合的「评测策略」、分场景评分细则、可替换的评委模型、可扩展的 AI 适配器——将来加新工具/新业务不用大改。

## 九、存储与成本（量级概念）

每次运行都把「输入 + 仿写结果 + 评分理由 + 用的评委」完整存下来，方便长期追踪。

- 一次跑 100 条样本 ≈ 1–1.5 MB
- 一个场景一年 ≈ 2 GB；将来复制到 3–5 个场景，线性增长
- 应对：3 个月前的数据归档冷存，热库只留近期

**结论**：量级不大，但会持续长，所以一期就把「输出归一化存储」「归档策略」设计好了。
