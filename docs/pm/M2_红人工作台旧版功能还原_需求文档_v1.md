## 一、需求背景

当前红人工作台已经迁入人物档案、产品库、千川仿写、价值观仿写、直播脚本仿写、成片预审、复盘和素材库等模块，但部分模块只完成了页面入口或基础能力，和旧版慧敏工作台的实际功能存在差异。

本轮不是重新开发一套工作台，也不是复制旧代码，而是完成一次“功能还原补齐”：

1. 前台业务步骤、核心输入、人工确认点、输出结果和模块间连接，以旧版可用功能为基准。
2. 后端继续使用新平台统一架构，不恢复旧版本地 JSON、工具自带密钥、前端直连外部服务等旧实现。
3. 所有功能仍以红人为上下文，支持多红人数据隔离，不写死“慧敏”。
4. 已有独立工具入口可以保留，但红人工作台内必须形成完整可用的内嵌模块。

## 二、产品目标

### 2.1 核心目标

运营进入某个红人工作台后，可以完成以下工作：

```text
维护人物档案
→ 选择或新建唯一当前商品
→ 使用人物档案和商品生成千川仿写初稿
→ 运营先人工修改
→ 系统自动逐轮预审和改写
→ 产出审核后的最好版本
→ 按需生成价值观脚本或直播开播方案
→ 管理脚本文档和视频素材
→ 按需做完整视频成片预审
→ 上传五类材料完成红人维度复盘
```

### 2.2 成功定义

本轮完成后，除 TikTok 外，旧版八个页签的核心功能均可在新版红人工作台中使用；人物、产品和脚本之间不再依靠运营重复复制资料；人工智能实际收到页面展示的全部关键输入。

### 2.3 目标用户

主要用户为运营。红人只参与外部拍摄协作，不直接操作本工作台。

## 四、统一业务规则

### 4.1 红人上下文

从红人工作台进入任一工具时，系统必须固定使用当前路由中的 `kol_id`，不得让运营再次选择达人。

供人工智能使用的红人上下文至少包含：

- 红人姓名。
- 原有人设全文 `persona`。
- 内容规划 `content_plan`。
- 基本身份 `background`。
- 真实经历 `experience`。
- 关系网 `relationships`。
- 独家经历 `unique_story`。
- 其他补充 `extra_notes`。

字段为空时跳过该段，不阻塞生成；红人不存在或已删除时必须阻止操作并给出明确提示。

### 4.2 当前商品

- 每个红人在同一时间最多有一个当前商品。
- 商品库为平台共享商品库，通过红人与当前商品的关联保持红人维度的选择。
- 当前商品从千川产品库选择；没有商品时可在工作台直接新建，再自动选中。
- 选择新商品时替换旧商品，不允许多选。
- 脚本工具中切换商品时，必须同步更新该红人的当前商品，不支持仅对当次生成生效的临时商品。
- `kol_active_products` 如继续作为内部存储，可以复用，但前台不得呈现为多选工具，后端必须保证每个红人最多一条有效关联。
- 脚本工具提交商品标识后，后端必须重新读取商品详情，不信任前端拼接的产品文本。

### 4.3 人工智能输入可追溯

每次生成或审核应在任务上下文中记录：红人标识、商品标识、原版脚本字数、使用的功能、模型和产出标识。不得把完整敏感素材写入普通操作日志。

### 4.4 页面风格

业务步骤和交互能力向旧版对齐，视觉继续使用新平台的设计变量、共享样式、图标和红人工作台外壳。不得引入 Tailwind，也不得另做一套旧版主题。

## 5.1 人物档案

### 当前判断

五分区展示、分区编辑、保存、取消和更新时间已经实现，页面本身无需重做。

### 本轮改造

1. 保留当前人物档案页面和五个字段。
2. 建立统一的人物上下文读取逻辑，供千川仿写、价值观仿写、直播脚本仿写和复盘复用。
3. 各工具不得只读取 `persona` 和 `content_plan` 而忽略五分区档案。
4. 独家经历必须进入人工智能上下文，不能继续漏读。

### 验收标准

- 修改五个分区中的任意字段后，重新生成脚本时能够从任务输入或测试替身中证明新内容已进入模型输入。
- 不同红人的档案不能串用。
- 空白分区不显示无意义占位文字，也不阻塞生成。

## 5.2 产品库和当前商品

### 当前问题

- 字段已经存在，但列表只展示少量字段。
- “主推机制”的页面提示错误地偏向成分和技术。
- 工作台首页支持多个在售商品。
- 千川仿写和直播脚本仿写没有真正读取产品库商品。

### 本轮改造

1. 产品列表恢复为可扫描的完整商品卡，展示所有非空字段：产品昵称、最主推卖点、可视化、主推机制、推荐来源、用户反馈、独家卖点、获奖荣誉、功效承诺和“只有我有”。
2. 保留搜索、分页、新建、编辑和软删除。
3. “主推机制”定义恢复为价格钩子或促销力度，例如买赠、破价、限时赠品。
4. “只有我有”选中后，所有相关脚本提示词都必须收到该约束。
5. 工作台首页将“目前在售商品”改为“当前商品”，只允许选择一个。
6. 当前商品区域必须支持“选择已有商品”和“新建商品并选中”。
7. 千川仿写、价值观仿写和直播脚本仿写均可读取同一产品库。

### 验收标准

- 一个红人选择商品乙后，原商品甲立即不再是当前商品。
- 刷新页面后当前商品保持一致。
- 产品卡的全部非空字段在不进入编辑弹窗的情况下可查看。
- 生成接口使用数据库中的最新商品字段；前端伪造旧卖点不能覆盖后端数据。
- 删除当前商品前必须解除或替换当前关联，并给出明确提示。

## 5.3 千川仿写

### 目标流程

```text
自动加载当前红人和当前商品
→ 上传或粘贴原版爆款脚本
→ 生成人工智能初稿
→ 暂停，由运营阅读初稿
→ 运营可选择直接预审，也可输入修改意见或直接修改初稿
→ 运营点击“开始逐轮预审”
→ 系统审核当前稿
→ 未通过则按审核意见做最小修改
→ 再次审核，最多四轮
→ 从所有审核版本中选择最好版本
→ 运营可继续微调，微调后再审核一次
→ 保存和导出最终稿
```

### 输入要求

- 当前红人的完整人物上下文。
- 当前商品全部字段。
- 原版爆款脚本。
- 运营修改意见（可选）。
- 每轮预审结果和上一版脚本。

对标素材为空不阻塞生成；本轮不做素材库自动注入。

### 自动预审规则

1. 复用当前千川脚本预审的规则和结构化结果，不再要求运营手工复制到另一个页签。
2. 评级顺序：通过优于小改，小改优于不通过。
3. 同评级时，必须修改项更少的版本优先；仍相同时，较新的版本优先。
4. 评级通过立即停止；否则最多审核四轮。
5. 某轮审核接口失败时，不丢失当前稿，停止自动循环并提示运营人工处理。
6. 自动修改只修审核点名的问题，不随意重写已经通过的内容。

### 页面要求

- 商品区域展示当前商品摘要，并提供“切换商品”。
- 原版脚本支持文件解析和直接粘贴。
- 时间线展示初稿、运营修改稿、每轮预审结果和自动修改稿。
- 最终区展示最好版本、最终评级、剩余问题和轮次用尽提示。
- 支持保存到产出中心、导出文本和办公文档格式。
- 独立千川脚本预审页签保留，作为运营临时审核其他脚本的工具。

### 关键修复

当前页面收集的 `productText` 没有进入生成请求。本轮必须用自动化测试证明：商品卖点、主推机制和“只有我有”确实进入人工智能模型输入，而不是只显示在页面上。

### 验收标准

- 没有当前商品时不能生成，并可直接进入商品选择或新建流程。
- 有当前商品后，不再要求重复上传产品卖点卡。
- 运营可以在初稿后直接启动自动审核，也可先修改或提交修改意见。
- 系统选出的最好版本必须经运营确认后才能作为正式文案保存，系统不自动认定为正式文案。
- 模拟“不通过、不通过、通过”时，系统准确完成三轮审核并停止。
- 模拟四轮均不通过时，系统交出最好版本并提示人工确认。
- 模拟第二轮比第四轮更好时，最终交付第二轮版本，而不是机械交付最后一版。
- 刷新或返回页面时，已经保存到产出中心的最终稿可在产出中心查看。

## 5.4 价值观仿写

### 产品决策

当前“提炼红人价值观关键词”流程不作为新版入口。价值观仿写必须恢复旧版四步流程。

### 目标流程

#### 第一步：输入爆款原文

- 单独填写并锁定开头。
- 粘贴完整爆款原文。
- 开头和全文均必填。

#### 第二步：选择产品

- 默认使用当前商品；如切换产品，同步更新该红人的当前商品，不作临时切换。
- 产品只用于推导情绪方向，最终脚本不得出现产品名和直接商品信息。

#### 第三步：推导情绪方向

- 人工智能根据商品和红人档案返回 2 至 3 个方向。
- 每个方向包含：焦虑型或诱惑型、短标题、说明、情绪锚点。
- 解析失败或服务失败最多重试三次，重试后仍失败则明确提示。

#### 第四步：生成脚本和报告

模型按固定结构输出：

- 原文结构分析：总字数、段落数和各段功能。
- 改写脚本：开头原样保留，段落数量和节奏对应，正文完全重写。
- 情绪检测报告：触发句、恐惧强度、诱惑强度、产品联想、开头核查和优化建议。

### 后处理

- 用旧版双字组合相似度算法计算改写稿和原文的相似度。
- 相似度不高于 35% 显示安全；高于 35% 且不高于 50% 显示接近安全线；高于 50% 显示需要继续改写。
- 支持脚本和报告两个页签。
- 支持多轮修改。
- 支持保存到产出中心并导出脚本加报告。

### 验收标准

- 生成稿第一句与锁定开头逐字一致。
- 生成稿中不出现所选商品名称。
- 产品不同，情绪方向应收到不同的产品输入。
- 结构分析、脚本、报告三个部分解析失败时有明确错误，不把原始标签文本直接当成成功结果。
- 相似度算法有独立单元测试，三个阈值区间均有测试。
- 当前价值观关键词提取页面不再作为工作台入口；如保留旧接口，不得影响新流程。

## 5.5 直播间脚本仿写

### 当前判断

七模块开播方案、对标文案锁定、多轮修改、字数控制和文本导出已经基本实现。本轮只补输入连接和旧版产品选择体验。

### 本轮改造

1. 自动绑定当前红人，不再次选择达人。
2. 默认读取当前商品并组装完整卖点卡。
3. 没有当前商品时，可从产品库选择或新建；选择或新建后必须存为该红人的当前商品。
4. 人工智能读取完整人物上下文，尤其是真实经历、关系网、独家经历和其他补充。
5. 保留三种卖点顺序：背书、机制、种草；机制、背书、种草；背书、种草、机制。
6. 保留对标直播间文案上传、粘贴、编辑、确认和解锁。
7. 保留七模块输出、多轮修改、超字数自动压缩和文本导出。

### 验收标准

- 有当前商品时进入页面即可看到商品信息，不要求再次上传。
- 切换当前商品后，生成输入同步变化。
- 生成请求中同时存在商品卖点、完整人物档案和对标直播文案。
- 最终讲解脚本字数不超过对标文案字数；自动压缩后仍保留七模块结构。
- 独立直播脚本页面继续可用，不因工作台内嵌改造而失效。

## 5.6 千川成片预审

### 产品决策

工作台中的“千川成片预审”恢复旧版完整视频分析，不以现有“八张关键帧加文案转录”作为等价实现。

### 目标流程

```text
上传原片
＋ 上传已剪辑成片
→ 后端安全接收并暂存
→ 人工智能读取两条完整视频
→ 流式输出分镜分析、三维评分和优化建议
→ 展示、保存和导出报告
```

### 页面要求

- 工作台页签从禁用改为可用。
- 两个视频上传卡片：原片和成片。
- 显示文件名、大小、上传和分析状态。
- 建议支持 `mp4` 和 `mov`，产品提示为建议 500MB 以内；实际服务端限制必须在页面明确展示。
- 分析过程流式显示状态，超时和服务失败不丢失用户已选文件信息。
- 报告支持保存到产出中心和导出办公文档格式。

### 新平台架构约束

- 前端不得直接调用 Gemini。
- Gemini 凭证、模型和提示词必须进入管理端统一配置，不在代码或环境文件中写死。
- 后端通过平台适配层调用 Gemini，并写人工智能调用日志。
- 大文件不得长期保存在应用服务器本地磁盘。可使用临时目录、对象存储或供应商文件接口，但分析完成或失败后必须按策略清理。
- 不允许静默退化为关键帧模式。完整视频能力不可用时，应明确提示配置或服务问题。
- 现有独立“千川剪辑预审”关键帧工具可以保留，不在本轮删除。

### 验收标准

- 工作台内可以上传两条视频并获得流式报告。
- 测试替身能够证明提交给模型的是两条完整视频文件或可读取的完整视频引用。
- 缺少凭证、文件过大、格式不支持、分析超时和供应商失败均有明确错误。
- 失败后临时文件和临时供应商文件按照清理策略处理。
- 每次调用在管理端日志中可追溯到用户、功能、模型、状态和耗时。

## 5.7 复盘

### 当前判断

当前版已具备红人维度列表、草稿、五类材料、流式分析、详情、删除、重新复盘、复制和导出，整体保留。

### 本轮改造

1. 修复“千川素材脚本”多文件解析：每个文件必须单独解析、保留原文件名和对应文本。
2. 不得把全部文本合并到第一条记录并让其余记录为空。
3. 复盘继续按当前红人隔离。
4. 人工智能分析时可读取完整人物上下文，但不能因为人物字段为空而阻塞复盘。
5. 至少有直播汇总数据或素材明细数据中的一种，才允许开始分析，沿用旧版要求。

### 验收标准

- 同时上传三份千川素材脚本后，数据库中三条文件名和文本一一对应。
- 删除或替换其中一份不影响其他脚本。
- 红人甲不能查看、修改或删除红人乙的复盘记录。
- 草稿、完成、重新复盘、复制和导出能力回归通过。

## 5.8 素材库

### 当前问题

当前素材库支持六类文本素材的添加、展开和删除，但缺少旧版文档上传解析、视频原片、在线播放和编辑。

### 本轮改造

1. 保留当前红人维度和六类素材结构，不退回固定慧敏或全团队混用。
2. 所有分类至少支持：标题、数据说明、脚本正文、可选脚本文档和可选视频原片。
3. 脚本文档支持上传解析，并允许运营在保存前修改解析出的正文。
4. 视频通过平台对象存储适配层上传，不存本地永久路径，不保存长期公开链接。
5. 列表显示标题、数据说明、分类、是否有视频和正文摘要。
6. 展开后可播放视频并查看完整正文。
7. 支持编辑标题、数据说明和正文；视频默认保持不变，换视频需执行明确的替换操作。
8. 删除使用软删除或业务删除策略，同时按对象存储规范处理视频对象；禁止直接物理删除数据库记录。
9. 播放地址使用后端鉴权或短时签名地址，不能把长期公开地址存入页面数据。

### 数据兼容

- 现有 `likes` 数值字段继续兼容。
- 新版需要支持旧版自由文本“数据/点赞数”说明；如需增加字段或调整表结构，必须先更新数据库和接口契约。
- 旧数据在没有文档或视频时继续正常展示。

### 验收标准

- 上传办公文档后，正文自动填充并可编辑。
- 上传视频后，刷新页面仍可在有权限的情况下播放。
- 编辑标题、数据说明和正文后即时生效，视频不被误删。
- 删除素材后普通列表不可见，其他红人也不可访问其视频。
- 对象存储上传、签名读取和删除操作均有服务日志。

## 七、新平台架构硬约束

开发必须同时遵循：

- 根目录 `AGENTS.md`。
- `docs/pm/项目开发工作流程与规范.md`。
- `docs/standards/功能迁移流程规范.md`。
- `backend/docs/后端开发约定.md`。
- `frontend/docs/前端规范.md`。
- `backend/docs/base/` 下接口、数据库和权限事实源。

### 7.1 后端

- 非流式接口返回统一响应信封；流式和文件下载按规范作为例外。
- 所有写操作记录操作日志。
- 人工智能、对象存储和其他外部服务必须走统一适配层和凭证池。
- 人工智能调用必须明确解析并传递供应商，不依赖默认值。
- 提示词和模型必须在管理端可配置，并支持现有红人专属提示词覆盖机制。
- 产出进入 `outputs`，长任务进入 `task_jobs`；不得只存在前端内存。
- 列表必须分页，删除遵循软删除要求。
- 若新增或修改数据表、接口、权限，先停止实现并向 PM 回报，再更新 Base 契约后继续。
- 新路由如使用 `AsyncSessionLocal`，同步加入测试会话替换清单。

### 7.2 前端

- 红人工作台继续使用 `KolWorkspacePage` 的外壳和内嵌模块模式。
- 普通 JSON 请求必须走 `request.ts`；仅流式、表单文件和文件下载可以使用原生请求，并手动附带鉴权信息。
- 使用现有设计变量和共享样式，不硬编码一套旧版颜色和间距。
- 使用 Ant Design 现有组件和图标；不手绘图标，不引入 Tailwind。
- 文件上传、流式生成、空状态、错误状态和重试状态必须完整。
- 独立工具路由不得因内嵌工作台改造而回归。

### 7.3 安全与数据隔离

- 所有工作台数据请求必须校验 `kol_id` 和资源归属。
- 禁止无鉴权读取素材视频和报告。
- 禁止前端持有第三方密钥。
- 禁止把临时视频、完整提示词或敏感素材写入普通日志。
- 用户不可访问其他无权限红人的素材和复盘数据。

## 九、测试和验收要求

### 9.1 测试方法

- 每个差异点先写失败测试，再实现功能。
- 人工智能和外部服务使用可控测试替身，验证收到的实际输入。
- 联合业务验收必须使用真实红人资料、真实商品、真实文件与视频，并通过平台已配置的真实人工智能服务验证输出质量；测试替身不能代替最终业务验收。
- 后端覆盖正常、校验失败、权限失败、外部服务失败和资源不存在。
- 前端覆盖主流程、禁用条件、错误提示、切换红人或商品后的状态更新。
- 文件和视频功能必须有真实格式的最小样本测试，不能只测空对象。

### 9.2 最小端到端验收链路

```text
运营登录
→ 进入红人甲工作台
→ 编辑人物档案
→ 新建并选中唯一当前商品
→ 千川仿写生成初稿
→ 运营修改
→ 自动预审和改写至通过或轮次用尽
→ 保存最终稿并在产出中心查看
→ 使用同一商品生成价值观脚本和直播开播方案
→ 上传脚本文档和视频到素材库
→ 上传五类复盘材料并生成报告
→ 上传原片和成片完成完整视频预审
→ 在管理端查看调用日志
```

### 9.3 回归范围

- 红人列表和红人工作台首页。
- 人物档案、产品库和当前商品。
- 种草仿写、人设仿写、直播复盘。
- 独立千川仿写、直播脚本、千川脚本预审和千川剪辑预审。
- 产出中心、管理端工具配置和调用日志。
- TikTok 工具保持现状，但不新增工作台入口。

### 9.4 联合验收节奏

本轮采用“连续开发、集中验收、集中修复、最终复验”的节奏，不在每个开发批次后等待 PM 回复。

#### 阶段一：连续开发

- 开发会话按批次一至批次五连续推进，不设常规 PM 停工检查点。
- 每个批次完成后，开发会话在任务文档中记录改动、测试证据、已知问题和下一批次计划，记录后直接继续。
- 中间进度信息仅做同步，不要求用户逐条确认。
- 只有出现业务规则无法判定、必须新增超出本文档的数据库或权限变更、外部服务无法使用、存在数据破坏或安全风险时，才中断相关模块并向 PM 提问；其他模块继续开发。

#### 阶段二：验收前准备

开发会话完成八个模块和集成回归后，一次性准备：

- 可稳定使用的验收环境和版本。
- 固定的测试红人、商品、脚本、文档、视频和五类复盘材料。
- 按本文档生成的逐项验收清单。
- 自动测试、真实浏览器测试、已知问题和未完成项清单。

存在阻断主流程或会导致数据错误的已知问题时，不发起联合验收。

#### 阶段三：第一次联合验收

- 预留 60 至 90 分钟，由用户和 PM 会话共同验收，开发会话提供环境和技术证据。
- 按第 9.2 节主流程完整操作一次，再按八个模块的验收标准补充检查。
- 验收期间只记录现象、预期和严重程度，不边验收边零散修改。
- PM 会话在验收结束后整理成一份问题包，一次性发给开发会话。

#### 阶段四：集中修复

- 开发会话按同一份问题包连续修复，不逐个问题等待 PM 确认。
- 涉及产品规则变更的问题单独标记，由 PM 会话集中找用户确认一次。
- 修复完成后运行全量回归，提供问题与修复结果对照表。

#### 阶段五：最终复验

- 预留 30 至 45 分钟，由用户和 PM 会话共同完成。
- 重点复验第一次未通过项，并快速重跑一次第 9.2 节主流程。
- 全部通过后由用户统一签收，再进入合并和上线决策。
