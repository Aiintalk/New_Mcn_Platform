# Git 工作流程规范

> 本文件是项目统一的 Git 操作规范。**所有开发——无论新功能还是功能迁移——完成后都必须走本流程推送到仓库，不得直接推 main。**
> 仓库：`https://github.com/Aiintalk/New_Mcn_Platform.git`
>
> **与 CLAUDE.md「⓪ 开工前准备」的分工（不重复）：**
> - CLAUDE.md ⓪：进项目时只"看清楚"——检查分支状态、读 PM 记忆/README 了解现状，**不拉代码、不切分支、不对齐数据库**。
> - 本文件：**开始一个功能时**才执行同步动作——`checkout main` → `pull` 拿最新 → 对齐数据库迁移 → 切功能分支。同步只在这里发生。

---

## 一、AI 的 Git 权限边界（重要）

| 操作 | 谁来做 |
|------|--------|
| 切分支 / 开发 / 提交(commit) / 推送分支(push) | ✅ AI 可执行 |
| 发起 PR（Pull Request） | ✅ AI 可执行 |
| 发 PR 成功后反馈（PR 链接/状态）给用户 | ✅ AI 必做 |
| **合并 PR 到 main** | ❌ AI 不做，由人工操作 |

- AI 推送分支、发起 PR 后，**反馈 PR 链接并提示"可以去合并了"**，然后停在这里。
- **合并到 main 由人工完成**（主干是最关键的闸门，留在人手里，防止把未审代码合进主干）。
- 发完 PR 后，开发**不必等合并**，可以按第三节规则切新分支继续做下一个功能。

**总原则——AI 做判断，风险操作人确认：**
AI 负责"检查、判断、给建议"；涉及风险的实际操作（切分支、`stash`、丢弃改动、解决冲突、撤销提交、合并）**必须先报告状态 + 给出处理选项，等用户确认后才执行**。AI 不擅自处理任何可能丢失代码或泄露密钥的操作。

## 二、分支命名

```
migrate/{功能名}    迁移旧功能（如 migrate/persona-positioning）
feature/{功能名}    开发新功能（如 feature/excel-export）
fix/{功能名}        修复 bug（如 fix/homepage-layout）
```

规则：
- 类型前缀三选一：`migrate/` / `feature/` / `fix/`
- 功能名小写英文 + 连字符（kebab-case），不用中文、空格、下划线
- **一个功能一个分支，雷打不动**（原因见第三节）
- `main` 是主干，受保护，不直接推

## 三、多功能并行规则（关键，避免两个功能绑死）

> 场景：A 做完功能1、推送发了 PR，但管理员今天没合并。A 要继续做功能2，该怎么办？

**规则：做下一个功能时，从最新 main 重新切新分支，不要在上一个功能的分支上继续开发。**

```bash
# 功能1 做完、推送、发 PR 后，要做功能2：
git checkout main
git pull origin main             # 拿最新（若管理员合并了别人的代码，这里能拿到）
git checkout -b feature/功能2     # 从 main 切全新分支
```

**为什么必须分开分支：**
- 若在功能1 的分支上继续做功能2，功能2 的提交会自动进入功能1 那个**还没合并的 PR**，两个功能绑死，管理员无法只合并功能1。
- 分开分支后，每个功能是**独立的 PR**，管理员可分别、按各自节奏合并，互不拖累。A 也不用等合并，切了新分支就能继续干。

**例外——功能2 依赖功能1（功能1 的代码功能2 要用）：**
```bash
git checkout feature/功能1
git checkout -b feature/功能2     # 从功能1切，带上功能1的代码
```
此时在功能2 的 PR 说明里标注"**依赖 PR #xx，需先合并它**"，管理员先合功能1、再合功能2。

## 四、标准流程（一个功能从开始到合并）

```
0. 检查分支状态     → AI 跑 git status + git branch 看清状态（见下方说明）
1. 同步 main        → git checkout main && git pull
2. 对齐数据库迁移   → 拉完代码后，确认本地数据库已含最新表（防"表不存在"，见下方说明）
3. 创建功能分支     → git checkout -b 类型/功能名（多功能并行见第三节）
4. 开发             → 在本分支完成开发（新功能 or 迁移）
5. 本地测试         → pytest / vitest / 覆盖率门禁，必须通过
6. 全量回归         → 确认没改坏其他功能
7. 更新文档         → 任务/验收/测试报告/README/PM记忆（见工作流程规范 节点 B++）
8. 提交             → git status 确认 → git add → commit（规范见第六节）
9. 推送分支         → git push origin 类型/功能名
10. 发起 PR         → AI 发 PR，反馈链接给用户
11. 人工合并         → 用户审核后合并到 main（AI 不合并）
12. 清理分支         → 合并后删除功能分支
```

**第 0 步 · 检查分支状态（AI 检查，人确认处理）：**
- AI 自动执行 `git status` + `git branch`，看清：当前在哪个分支、是不是 main、工作区有没有未提交改动。
- 若状态不干净（不在 main / 有未提交改动）→ AI **列出情况 + 处理选项**（回 main / 提交到原分支 / `stash`），**等用户确认后才操作**，不擅自处理（避免弄丢未提交的代码）。
- 状态干净且在 main 后，才进入第 1 步。

**第 2 步 · 对齐数据库迁移（拉完代码后）：**
- 查看 `backend/scripts/init_db.sh` 与 `backend/migrations/`，确认本地数据库已含最新表。
- `init_db.sh` 可安全重复运行就跑它；否则只按顺序补执行缺失的迁移。
- 防止"拉了新代码但本地数据库没更新 → 前端写入报表不存在"。

> 与旧流程的区别：**不再部署测试服**，开发时本地测试通过即可（pytest/vitest/覆盖率门禁）。

**开发中可随时阶段性保存**：一个功能做几天/跨多个会话时，可随时 `commit + push` 到自己分支（防丢、留进度），功能完成后再发 PR。

## 五、详细步骤与命令

### 1. 同步 main + 创建分支

```bash
git checkout main
git pull origin main
git checkout -b 类型/功能名        # 替换成实际分支名，如 feature/excel-export
```

### 2. 开发 + 本地测试

```bash
# 后端
cd backend && source .venv/bin/activate
pytest tests/ -v
python scripts/run_coverage.py --gate

# 前端
cd frontend
npx vitest run --coverage
```

**测试不通过不许进入下一步。** 必须本地全绿 + 全量回归通过。

### 3. 提交 + 推送（提交前先看一眼，防误提交）

```bash
git status                        # 先确认改了哪些文件，别误提交 .env / 临时文件
git add <具体文件>                 # 按需添加；用 git add . 时务必先看过 status
git commit -m "feat: 实现红人列表 Excel 导出"
git push origin 类型/功能名
```

> ⚠️ **安全**：绝不提交 `.env`（含数据库密码、API Key）。确认 `.gitignore` 已排除 `.env`、`.venv`、`node_modules`、`__pycache__` 等。一旦密钥误提交到远程，等于泄露。

### 4. 发 PR + 人工合并

- AI 在 GitHub 对该分支发起 PR，目标分支 `main`，填写说明（做了什么、测试结果、影响范围、是否依赖其他 PR）。
- **AI 发 PR 成功后，反馈 PR 链接给用户，提示可以合并，然后停下。**
- 用户审核 → 合并到 main。
- 合并后删除功能分支：`git branch -d 类型/功能名`（本地）+ GitHub 上删远程分支。

## 六、Commit 提交规范

格式：`类型: 简短描述`

| 类型 | 用于 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `migrate` | 功能迁移 |
| `docs` | 文档变更 |
| `test` | 测试相关 |
| `refactor` | 重构（不改功能） |
| `chore` | 杂项（依赖、配置） |

示例：
```
feat: 实现红人列表 Excel 导出
fix: 修复运营首页统计卡片不刷新
migrate: 迁移人格定位功能到新架构
```

要点：一次提交只做一件事；描述说清"做了什么"。

## 七、两人协作铁律（防覆盖、防冲突）

> 两人各自开分支做不同功能，平时互不影响；只在合并到 main 时统一处理冲突。

1. **各自开分支，不在同一分支互相推拉。**
2. **拉别人代码前，先 commit 自己的改动。**
   - 已提交的代码，git 永远不会让它凭空丢——最坏是冲突，冲突可解决。
   - 未提交就去 pull，碰到同一文件才可能丢改动。
   ```bash
   git add . && git commit -m "..."
   git pull origin main
   ```
3. **冲突不可怕。** pull 出现冲突时 git 会停下让你选/合，不会自动覆盖；手动解决后提交，双方改动都在。
4. **系统文件冻结区的改动单独提交、并知会对方**（见第八节）。

---

## 八、系统文件冻结区（改动需特殊对待）

> 公共核心服务被多个功能共享，改坏它会波及一片。开发/迁移时最危险的就是为了让本功能跑通，顺手改了公共服务。

**冻结区文件**（具体清单以《功能迁移流程规范》第四节为准，待 Claude Code 读真实代码核准）：
- 后端：`app/core/`、`app/middlewares/`、`app/adapters/`、统一响应/错误码、公共表、`migrations/`
- 前端：`src/api/request.ts`、`src/styles/variables.css`、`src/store/`、`src/routes/`、`src/layouts/`
- 跨功能：路由表、菜单注册、密钥池、模型配置

**改动冻结区的规矩：**
1. 优先不改——能在本功能自己的文件里解决就不动它。
2. 非改不可时：先全局搜索引用、列出影响哪些功能。
3. **知会另一个协作者**（他可能正用着）。
4. 改完跑**全量回归测试**，不只测当前功能。
5. **冻结区改动单独提交**（独立 commit/PR），不混在功能开发里，便于追溯和回滚。

## 九、异常处理

| 情况 | 处理 |
|------|------|
| pull 出现冲突 | git 停下提示冲突文件，手动编辑解决（保留双方需要的改动）→ `git add` → `git commit` |
| 本地有未提交改动又想 pull | 先 `git commit`（或 `git stash`），再 pull |
| 误提交了 .env / 密钥 | 立即停下告知用户——需从历史中移除并轮换密钥，不可只是删文件再提交 |
| 提交错了分支 | 先停下问清楚，可用 `git cherry-pick` 挪到对的分支 |
| 撤销最后一次提交（未推送） | `git reset --soft HEAD~1`（保留改动） |
| PR 有冲突无法合并 | 在功能分支 `git pull origin main` 合入最新 main，解决冲突后再推 |

> 拿不准的 git 操作（尤其 reset / rebase / force push / 删历史）先停下来问，不要凭感觉操作，避免丢提交或泄露密钥。
