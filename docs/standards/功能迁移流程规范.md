# 功能迁移流程规范（旧架构 → 新架构）

> 适用：将旧架构（Ai_Toolbox）的功能板块逐个迁移到新架构（MCN Platform）。
> **本规范与《项目开发工作流程与规范》（docs/pm/项目开发工作流程与规范.md）必须同时遵循，缺一不可。**
> - PM 工作流程 = 骨架：迁移仍走六节点流程（需求确认 → 拆解评审 → 开发 → 测试验收 → 文档落地）。
> - 本规范 = 迁移专属约束：在上述流程之上，额外满足迁移红线、检查清单、冻结区、DoD。
>
> **用户提出功能迁移需求时，必须同时读取这两份文件并一起执行。** 红线会持续增加，每次迁移以本文件最新内容为准。
> 核心理念：迁移 ≠ 复制粘贴。旧功能要**长成新架构的样子**——融入统一的入口、服务、配置、日志体系，而不是把旧代码原样塞进来。

## 一、迁移红线（硬约束，可持续增加）

> 每条都是"不满足即不算迁完、不得推送 GitHub"的红线。本清单会随迁移推进持续补充。

| # | 红线 | 说明 |
|---|------|------|
| 1 | **运营端入口必须在「创作中心」** | 不新增顶级侧边栏菜单，遵循运营端布局规范 |
| 2 | **运营端产出必须在「产出中心」显示** | 功能跑出的结果要进产出中心，不能只在功能页内自存 |
| 3 | **公共服务必须用系统架构内的统一服务** | AI / TikHub / OSS / ASR 一律走新架构密钥池+适配层，**禁止旧功能自直连或自带一套** |
| 4 | **Prompt + 模型调用必须在管理端可配置** | 旧功能写死的 Prompt、模型选择，迁后上提到管理端「服务配置」，**禁止硬编码在代码里** |
| 5 | **功能板块必须纳入管理端「功能配置」** | 可管理上线开关（dev → online），控制运营端是否可见 |
| 6 | **调用第三方服务必须产生日志** | AI/TikHub/OSS/ASR 每次调用写 `external_service_logs` / `ai_call_logs`，管理端调用日志页可见 |
| 7 | **拿不准时参照已迁功能或 README** | 以已迁好的功能（如 `persona-positioning`）为样板，或读 README 理解 |
| … | （后续红线在此追加） | |

## 二、迁移检查清单（每迁一个功能，逐项打勾）

> **使用说明**：每个 `【闸门】` 标记的步骤，必须等到条件满足后才能继续。
> 标注 `Skill:` 的步骤必须实际调用对应技能，不得跳过或用其他方式替代。

### A. 迁移前（需求确认阶段）

**第一步：启动**
- [ ] `Skill: superpowers:using-superpowers` — 确认技能可用
- [ ] 读取本规范文档 + `docs/pm/项目开发工作流程与规范.md`

**第二步：需求澄清**
- [ ] 本功能依赖哪些外部服务？（AI / TikHub / OSS / ASR）
- [ ] 是否需要改基础服务/系统文件？若是 → 走第四节「系统文件冻结区」流程
- [ ] 定 `tool_code`（如 `persona-positioning`）
- [ ] 参照已迁功能样板，明确入口/产出/配置的标准做法
- [ ] **红线 4 明确确认**：Prompt 和模型是否硬编码？若有疑问必须停下来问用户，不得默照旧代码
- [ ] 逐条列出假设和疑问，向用户复述理解的需求
- [ ] 【**闸门**】用户**文字回复确认**需求无误 → 未确认不得进入下一步

**第三步：拆分方案**
- [ ] `Skill: superpowers:writing-plans` — 产出前后端任务拆解方案
- [ ] 【**闸门**】用户**审阅并批准**拆解方案 → 未批准不得开始写代码

---

### B. 后端层（开发阶段）

每个后端子任务开发前：
- [ ] `Skill: superpowers:test-driven-development` — 先写测试（红），再写实现（绿）
- [ ] 服务对接：走新架构统一 adapter（不自直连）→ 对应红线 3
- [ ] Prompt / 模型配置迁移：上提到管理端可配 → 对应红线 4
- [ ] 接口实现 + 在 Base_API 登记契约
- [ ] 数据表（若需要）+ 在 Base_Database 登记
- [ ] 第三方调用写日志 → 对应红线 6
- [ ] `Skill: superpowers:requesting-code-review` — 子任务实现完成后做 review
- [ ] 【**闸门**】review 通过，该子任务才算完成

---

### C. 前端运营端（开发阶段）

每个前端子任务开发前：
- [ ] `Skill: superpowers:test-driven-development` — 先写测试（红），再写实现（绿）
- [ ] 功能入口放「创作中心」，不新增顶级菜单 → 对应红线 1
- [ ] 工具卡片（绑定 `tool_code`）
- [ ] 功能页面
- [ ] 首页「常用工具」快捷入口（如适用）
- [ ] 产出接入「产出中心」 → 对应红线 2
- [ ] `Skill: superpowers:requesting-code-review` — 子任务实现完成后做 review
- [ ] 【**闸门**】review 通过，该子任务才算完成

---

### D. 前端管理端

- [ ] 纳入「功能配置」，含上线开关 dev→online → 对应红线 5
- [ ] 相关服务配置入口可用（AI / TikHub Tab）→ 对应红线 4

---

### E. 上线前提

- [ ] AI Key 配好 + 绑定模型（若依赖 AI）
- [ ] TikHub Key 配好（若依赖抖音号解析等）
- [ ] 其他依赖服务 Key 配好（OSS/ASR 若依赖）
- [ ] 工具状态从 dev 改为 online

---

### F. 测试 + 推送

- [ ] `Skill: superpowers:verification-before-completion` — 声称完成前必须调用，提供可验证证据
- [ ] 本地测试通过（pytest / vitest / 覆盖率门禁）
- [ ] **全量回归**：跑全量测试，确认没改坏其他已迁功能 / 基础服务
- [ ] 文档落地：需求文档 / 后端任务单 / 前端任务单 / 测试报告 / Base_API 更新 / PM 记忆更新
- [ ] git 提交 + PR（见第五节）
- [ ] 【**闸门**】以上全部完成，才能向用户声称「迁移完成」

## 三、单功能"迁完"的定义（DoD）

一个功能满足以下全部，才算迁移完成、可推送：

```
1. 二（A~F）检查清单全部勾选
2. 一、迁移红线全部满足（无任一违反）
3. 本地测试通过 + 全量回归通过（旧功能没被改坏）
4. 文档落地（含功能位置总结）
5. git 在功能分支、PR 已发起
```

## 四、系统文件冻结区（防止改坏基础服务）

> 迁移功能时最危险的是为了让本功能跑通，改了多功能共享的基础服务，悄悄改坏其他功能。

**冻结区文件（改动需特殊流程）**——以下为推断清单，**待 Claude Code 读真实代码核准后替换**：

```
后端核心：
  app/core/          配置/数据库/响应封装/安全/seed
  app/middlewares/   鉴权中间件
  app/adapters/      AI/TikHub/OSS/ASR 适配器（多功能共享）
  app/core/response.py  统一响应/错误码
  app/models/ 公共表（users/credentials 等）
  backend/migrations/   数据库迁移

前端核心：
  src/api/request.ts      请求封装
  src/styles/variables.css 设计 token
  src/store/              全局状态
  src/routes/             路由守卫
  src/layouts/            公共布局

跨功能配置：
  路由表 / 菜单注册 / 密钥池 / 模型配置
```

**改动冻结区的规矩：**
1. **优先不改** —— 能在本功能自己的文件里解决就不动冻结区。
2. **非改不可时**：先全局搜索引用、列出影响哪些功能。
3. **知会另一个协作者**（他可能正用着）。
4. **改完跑全量回归测试**，不只测当前功能。
5. **冻结区改动单独提交**，不混在功能迁移里。

> 普通功能文件（`app/routers/` 某功能路由、`app/services/` 某功能服务、`src/pages/` 某页面）可正常改，只测当前功能即可。

## 五、Git 协作流程（一功能一分支，本地测试，PR 合并）

> 详细分步见《GIT_WORKFLOW.md》。要点：

1. 从最新 main 切分支：`git checkout -b {日期}_{功能名}`（如 `0611_persona-writer`）。
2. 在本分支迁移（不在 main 上直接改）。
3. 本地测试通过（pytest/vitest/覆盖率门禁）+ 全量回归。
4. **拉别人代码前先 commit**（已提交的永不丢，最坏是冲突可解决）。
5. 更新文档（README/CHANGELOG/PM 记忆/功能位置总结）。
6. 提交 → 推分支 → 发 PR → 审核 → 合并 → 清理分支。
7. 两人各在各的分支迁不同功能，避免撞车；改公共文件遵循第四节冻结区规矩。

## 六、兜底规则

迁移中任何决策拿不准（入口放哪、配置怎么接、服务怎么调、日志怎么写）：
1. **看已迁移的新功能**（如 `persona-positioning`）是怎么做的，照样板做。
2. **读对应端的 README / docs**（backend/docs、frontend/docs）理解架构约定。
3. 仍不确定 → 停下来问用户，不擅自决定。
