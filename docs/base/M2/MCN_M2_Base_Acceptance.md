# MCN Information System Platform · M2 Base Acceptance 验收标准

> 文档定位：本文件定义 M2 阶段功能验收标准。M1 验收见 `docs/base/M1/MCN_M1_Base_Acceptance.md`。

---

## 1. M2 Sprint 1 — kol-intake 红人入驻问卷

### 1.1 管理员后台

| # | 验收项 | 验收方法 |
|---|--------|----------|
| A1 | 管理员可查看并编辑 24 道题目（顺序、内容、启用状态） | 进入「入驻问卷 → 题目管理」，拖拽排序后刷新页面，顺序保持 |
| A2 | 管理员可配置对话 AI 和报告生成 AI（分别选模型 + 填 system_prompt） | 进入「入驻问卷 → AI 配置」，保存后重新进入页面，配置持久化 |
| A3 | 管理员可查看全部博主的提交记录和报告 | 进入「入驻问卷 → 提交记录」，可看到所有 operator 的数据 |

### 1.2 运营端

| # | 验收项 | 验收方法 |
|---|--------|----------|
| B1 | 运营可为博主生成分享链接，链接有效期可设置 1-720 小时 | 生成链接，检查 `expires_at` 正确，有效期外访问返回 410 |
| B2 | 链接列表展示正确的博主姓名、过期时间、访问/提交状态 | 生成多条链接，博主访问和提交后刷新列表，状态正确更新 |
| B3 | 运营下载报告需满足「链接已过期或博主已下载」条件 | 博主未下载且链接未过期时，运营下载返回 403，提示信息正确 |
| B4 | 运营只能看到自己生成的链接和提交记录 | 用两个 operator 账号交叉验证，互相看不到对方数据 |

### 1.3 博主端（公开页面，无需登录）

| # | 验收项 | 验收方法 |
|---|--------|----------|
| C1 | 博主打开链接后，AI 自动发送开场白 | 用有效 token 访问 `/intake/{token}`，页面加载后立即显示 AI 开场白 |
| C2 | AI 对话引导覆盖 24 道题目的核心内容 | 完整对话后检查报告，关键信息字段均有填充 |
| C3 | 博主提交后显示「报告生成中」状态，并可轮询查询 | 提交后页面显示 loading，轮询 `/status` 直到 `ready` |
| C4 | 报告生成完成后，博主可下载 docx 和 pdf | 报告 ready 后点击下载，两种格式均可正常打开 |
| C5 | 重复访问已提交的链接，显示「已提交」状态和历史对话（只读） | 用已提交的 token 再次打开，不允许再次提交 |
| C6 | 过期链接访问提示已过期（410） | 手动将 `expires_at` 改为过去时间，访问返回 410，页面友好提示 |

### 1.4 非功能

| # | 验收项 | 验收方法 |
|---|--------|----------|
| D1 | AI 对话响应延迟 < 5 秒（95th percentile） | 压测 10 次对话，记录响应时间 |
| D2 | 报告生成时间 < 60 秒 | 记录从 submit 到 `report_status=ready` 的时间 |
| D3 | 并发 5 个博主同时对话不报错 | 同时开 5 个浏览器窗口，各自进行对话 |
| D4 | 报告文件可正常打开，格式正确（无乱码，排版正常） | 下载 docx/pdf，用 Word 和浏览器分别打开验收 |

## 2. M2 Sprint 2 — 运营首页重设计

### 2.1 功能

| # | 验收项 | 验收方法 |
|---|--------|----------|
| E1 | 首页顶部展示 4 张统计卡片：今日任务数、总产出数、签约红人数、在线工具数 | 登录 operator 账号，检查 4 卡片数值与数据库实际数据一致 |
| E2 | 折线图展示近 7 天任务数和产出数趋势 | 切换 7 天/30 天，图表数据随之变化 |
| E3 | 饼图展示各工具使用占比 | 使用不同工具后刷新，占比按实际任务数更新 |
| E4 | 快捷入口卡片展示在线工具，点击跳转正确 | 点击每个快捷入口，路由跳转到对应工具页 |
| E5 | 近期任务表格和近期产出表格数据正确 | 对比任务列表页，首页表格显示最新 5 条 |

### 2.2 权限

| # | 验收项 | 验收方法 |
|---|--------|----------|
| F1 | operator 统计卡片和图表只统计自己的数据 | 两个 operator 账号登录，数据互不干扰 |
| F2 | admin 不进入此页面（admin 有独立首页） | admin 账号无「概览」菜单项 |

### 2.3 UI

| # | 验收项 | 验收方法 |
|---|--------|----------|
| G1 | 页面在 1280px 和 1920px 宽度下布局正常，无遮挡 | 调整浏览器窗口宽度验证 |
| G2 | 品牌色 `#f59a23` 正确应用于卡片 icon、图表色彩 | 截图对比设计稿 |
| G3 | 图表加载中显示骨架屏或 spin，数据加载完成后正常渲染 | 网络降速后刷新页面验证 |

## 3. 验收流程

```
1. 后端部署最新代码到测试环境
2. 前端部署最新代码到测试环境
3. 运行 migration 确认 4 张新表存在
4. 按 Sprint 1 验收项逐条测试（A → D）
5. 按 Sprint 2 验收项逐条测试（E → G）
6. 记录不通过项，提 bug 单给对应端修复
7. 修复后回归验收，全通过后 PM 签收
```

---

## 4. 不在 M2 验收范围内

- kol-intake 题目的 AI Prompt 优化（内容质量由运营持续迭代）
- 报告模板样式细节（M2 只验收能正常生成和下载，样式迭代后续 Sprint）
- OSS 存储（M2 存本地，M3+ 再迁移）
- 统计数据的精确性（M2 统计为近似实时，不做精确对账）
